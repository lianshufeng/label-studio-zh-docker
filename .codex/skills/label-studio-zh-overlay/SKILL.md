# Label Studio 中文 Overlay

## 核心目标

1. 目标不是临时修几个中文词，而是持续维护一套“基于官方正式版源码、可重复升级、可直接构建运行”的中文 overlay 方案。
2. 交付物必须是正式仓库文件，而不是一次性分析脚本或临时说明。
3. 主流程必须是“正式版确认 -> 正式产物骨架 -> 源码全量扫描 -> 结构化入库 -> 去噪提纯 -> 按模块整理词典 -> 注入验证”，不得跳步。
4. 运行时词典必须服务于 UI 中文化，不得把测试代码、样式片段、路径常量、第三方代码碎片当成 UI 文案。

## 临时目录约定

1. 下载、解压和分析最新版 Label Studio 源码时，统一使用仓库根目录 `.tmp/` 作为临时目录。
2. `.tmp/` 仅用于缓存和分析，禁止作为正式项目文件或发布产物目录。
3. 仓库必须在 `.gitignore` 中忽略 `/.tmp`，避免临时文件进入版本控制。

## 基本规则

1. 先确认当前工作区是否为 Git 仓库。
2. 先联网到 GitHub 官方仓库 `HumanSignal/label-studio` 的 Releases 页面查找最新正式 release；如需交叉核对，只能补充查看同仓库 Tags，不得用 Docker Hub、第三方镜像站或搜索摘要反推“最新版”。
3. 在用户仓库里创建或复用与版本号一致的分支，例如 `1.23.0`。
4. `master` 用于最终汇总后的最新版本，并可对应 Docker `latest`。
5. 不要求用户手动输入版本号，默认自动查 GitHub 最新正式 release；只有用户明确指定版本时才覆盖默认行为。
6. 再查 Docker Hub 对应标签。
7. 如果用户明确指定正式版本号，例如 `1.23.0`，默认直接使用同名 semver 标签 `heartexlabs/label-studio:1.23.0`，不得擅自改成 `ls-release-*`、RC、nightly、develop 或其他派生标签。
8. 只有在“用户未指定标签形式且同名 semver 标签确实不存在”时，才允许退回使用与 release 对应的 `ls-release-*` 标签；一旦退回，必须在回复中明确说明原因。
9. 不得把“我推测 release 镜像更稳”当作改用 `ls-release-*` 的理由；需要以用户要求和实际标签存在性为准。

## 必须产物

1. `Dockerfile`
2. `docker-compose.yml`
3. `patches/apply_overlay.py`
4. `overrides/static/zh-overlay/ls-zh.js`
5. `overrides/static/zh-overlay/ls-zh.css`
6. `README.md`
7. `.tmp/` 下的结构化扫描结果文件

## 必须流程

1. 先确认正式版本号和镜像标签策略。
2. 再生成或更新正式产物骨架。
3. 再下载并分析对应正式版源码。
4. 先把全量扫描结果写入结构化 JSON。
5. 对扫描结果做去噪和模块归类。
6. 基于模块化结果生成或更新运行时词典。
7. 再做模板注入、构建验证和回读校验。

## 禁止行为

1. 禁止把“升级到最新版”退化成只改 README、只改 Docker 标签或只改分支名。
2. 禁止在没有结构化扫描结果文件的情况下宣称词典已重生成。
3. 禁止跳过去噪，直接把源码扫描原始结果塞进 `ls-zh.js`。
4. 禁止把 `latest`、`develop`、`nightly`、RC、临时构建标签当默认正式版镜像。
5. 禁止整文件覆盖官方模板或直接修改官方前端源码。
6. 禁止把“看到页面漏一个补一个”当主流程，只能作为全量扫描之后的补充校对。

## 版本确认

1. 先查 GitHub 官方仓库 `HumanSignal/label-studio` 的 Releases 页面，确认标记为 `Latest` 或列表顶部的最后一个正式 release 版本号。
2. 如页面展示异常或需补充核对，只能继续查同一 GitHub 官方仓库的 Tags；若 Releases 与 Tags 不一致，以 Releases 为准。
3. 再查官方 Docker Hub `heartexlabs/label-studio` tags 页面，寻找与该 release 对应的标签；这一步仅用于确认可用镜像标签，不用于判断最新版。
4. 用户明确要求“直接使用与正式版同名的 Docker semver 标签”时，必须优先遵从该要求，不再自行切换到其他标签形式。
5. 如果 Docker Hub 没有干净 semver 标签，且用户没有明确禁止，才允许使用带 `ls-release-*` 的 release 对应标签。
6. 不选 `latest`，也不默认选临时分支构建标签。
6. 在回复中明确说明：
   - 采用的正式版本号
   - 数据来源链接
   - 为什么没有使用 `latest`

## Overlay 约束

1. 不修改官方前端源码。
2. 优先使用 Django 模板打补丁，不直接整文件覆盖模板。
3. 先识别实际模板继承链：
   - 主应用通常走 `base.html`
   - 登录和轻页面通常走 `simple.html`
4. 优先在公共入口模板打补丁，保证每个页面都具备统一的 hook JS 和 CSS。
5. 只在允许的 block 或稳定锚点中注入 CSS 和 JS。
6. JS 默认替换 UI 文案；对官方内置示例内容可条件翻译；对真实任务数据和业务载荷默认不翻译。
7. 保持最小改动，不整文件重写。

## 推荐流程

1. 自动查询 GitHub 最新正式 release。
2. 在用户仓库中创建或复用同名版本分支；如果分支已存在，则直接切换并在该分支上继续更新。
3. 先生成或更新“正式产物骨架”，至少包括：
   - `Dockerfile`
   - `docker-compose.yml`
   - `patches/apply_overlay.py`
   - `overrides/static/zh-overlay/ls-zh.js`
   - `overrides/static/zh-overlay/ls-zh.css`
   - `README.md`
4. 按 Docker overlay 方案生成或更新 `Dockerfile`，优先采用技能附件中的模板骨架：
   - `FROM heartexlabs/label-studio:<version>`
   - `COPY overrides/static/zh-overlay/ls-zh.js ...`
   - `COPY overrides/static/zh-overlay/ls-zh.css ...`
   - `COPY patches/apply_overlay.py /tmp/apply_overlay.py`
   - `RUN python /tmp/apply_overlay.py && rm -f /tmp/apply_overlay.py`
5. 下载该版本源码到仓库根目录 `.tmp/`，仅用于分析，不修改官方源码。
6. 先从源码中系统提取英文文案，再用运行页面做补充校验。
7. 必须优先运行 `scripts/extract_label_studio_strings.py` 对最新版源码做全量扫描，并把结果先写入 `.tmp/` 下的结构化 JSON 文件，不允许跳过“先入库”。
8. 必须基于扫描结果按模块整理 `ls-zh.js`，至少包含：
   - `auth`
   - `home`
   - `projects`
   - `import_export`
   - `data_manager`
   - `labeling`
   - `settings`
   - `organization`
   - `templates`
9. 使用 Docker 构建阶段补丁脚本，在模板稳定锚点插入 hook JS/CSS。
10. 重新构建并验证登录页、首页、主应用页和创建项目弹窗。

## 失败降级

1. 如果当前环境没有本机 Python，必须切换到容器内 Python 完成扫描，不允许跳过扫描。
2. 如果 Docker Hub 或外部网络异常，只能把问题标记为“外部依赖阻塞构建验证”，不能因此跳过源码扫描、词典生成或编码校验。
3. 如果扫描结果噪声过多，本次可以先完成“结构化入库 + 去噪规则更新 + 模块化词典骨架”，但必须明确说明尚未完成最终提纯，不能宣称已达到完整覆盖。
4. 如果只验证了注入链路，必须明确写“仅完成注入链路验证”，不能写成“升级已全部完成”。

## 执行兜底

1. 如果宿主机缺少可用 Python，必须改用容器内 Python 或临时 Python 容器执行 `scripts/extract_label_studio_strings.py`。
2. 不允许因为本机环境缺失而退化为手工维护少量高频词典。
3. 在无法完成源码全量扫描前，最多只能说明“注入链路已验证”，不得宣称 `ls-zh.js` 已完成重生成。
4. 如需手工补充词条，必须建立在全量扫描结果之上，只能作为补充校对，不能替代扫描生成。
5. 如果扫描结果里混入测试代码、样式类名、颜色值、路径、构建常量、第三方代码碎片，必须继续过滤提纯；不能把明显的噪声直接当 UI 词条塞进运行时词典。

## 重复执行

1. 技能必须支持同一版本重复执行。
2. 如果 GitHub 最新版本未变化，例如仍为 `1.23.0`，则复用现有版本分支，不重复创建新分支。
3. 生成 `ls-zh.js` 时应基于源码重新扫描并按稳定顺序更新，避免重复追加相同词条。
4. 模板补丁必须保持幂等，重复构建时不能重复插入同一段 CSS、JS 或配置脚本。
5. 临时下载目录固定为仓库根目录 `.tmp/`，可覆盖刷新或清理后重建，但不能污染正式项目目录。
6. 若本次执行未发现新版本或无新增词条，也应保持结果稳定，不报错、不重复写入。

## 字典生成方法

1. 必须优先从最新正式版源码中全量收集英文 UI 文案，包括：
   - Django 模板
   - 前端组件
   - 路由页面
   - 菜单项
   - 标题
   - 按钮
   - 说明文案
   - 备注文案
   - 模板分类
   - 模板标题
   - 表单标签和提示语
   - 空状态、帮助信息、卡片文案、弹窗文案
2. 容器内 Django 模板和运行页面 HTML 只用于补充校验，不替代源码提取。
3. 全量扫描结果必须先入库，再按模块整理，不采用只挑高频词的方式代替全量扫描。
4. 重点优先校对以下模块：
   - 登录注册
   - 首页欢迎区和资源卡片
   - 项目列表
   - 创建项目
   - 导入导出
   - 数据管理
   - 标注页按钮
   - 设置页
   - 成员与组织
   - 模板分类
   - 模板标题
5. 词典优先使用源码中的精确短语映射。
6. 对存在多空格、换行的 UI 文案，增加归一化匹配。
7. 按三层策略控制翻译范围：
   - 必翻：标题、按钮、菜单、说明、提示、空状态、帮助文案、弹窗文案、表单标签
   - 条件翻译：官方内置模板中的示例内容、固定演示文案、非用户输入的预览说明
   - 默认不翻：真实任务数据、标注对象原文、字段值、动态业务内容、可能影响标注语义的原始文本
8. 对代码块、脚本、样式、真实输入内容和真实任务数据区域继续保持跳过。
9. 不采用“看到页面漏一个补一个”或“只整理常见词汇”的方式作为主流程，优先通过源码全量扫描批量生成和维护字典。
10. `ls-zh.js` 的运行时词典必须按模块维护，不允许长期保持单一大平铺对象；至少要能看出来源模块和后续补词入口。
11. 生成词典前必须先做“去噪”，至少排除：
   - 测试用例字符串
   - CSS 类名
   - 颜色值
   - 文件路径
   - import/export 语句碎片
   - 第三方库代码片段
   - 构建产物和打包残片
12. 只有通过去噪后的字符串，才允许进入运行时词典。

## JS 实现要求

1. 使用 `MutationObserver` 处理 React 动态更新。
2. 使用 `TreeWalker` 遍历文本节点。
3. 跳过：
   - `SCRIPT`
   - `STYLE`
   - `CODE`
   - `PRE`
   - `TEXTAREA`
   - class 包含 `task-text` 或 `annotation` 的区域
4. 同时处理常见属性：
   - `placeholder`
   - `title`
   - `aria-label`
   - `value`
5. Hook `history.pushState`、`history.replaceState` 和 `popstate`。
6. 在实现层面为“条件翻译”的示例区域预留独立选择器或规则，避免和真实任务数据共用同一套替换逻辑。

## 验证方式

1. 构建并启动容器。
2. 检查容器内模板补丁结果和静态资源是否存在。
3. 访问 `/user/login/`、`/`、主应用页和创建项目弹窗，确认是否命中注入资源。
4. 若部分页面未命中，优先回查模板继承链，继续补公共入口模板，而不是直接整文件覆盖某个页面模板。
5. 抽样比对源码字符串、生成字典和运行页面，确认字典不是凭经验手写遗漏。
6. 回读所有修改文件，确认中文未乱码且为 UTF-8 无 BOM。
7. 若因外部网络、镜像仓库或环境问题导致构建失败，必须明确区分“代码问题”和“外部依赖问题”，不得把外部超时误报成技能执行完成。

## 验收门槛

1. 没有结构化扫描结果文件，不算完成。
2. `ls-zh.js` 不是基于扫描结果按模块整理，不算完成。
3. 只更新 README 或只改 Docker 标签，不算完成“升级到最新版”。
4. 使用了 `latest`、develop、nightly、RC 或与用户要求不一致的镜像标签，视为未按要求执行。
5. 中文乱码、非 UTF-8 无 BOM、整文件粗暴覆盖、未校验回读，均视为未完成。
6. 未说明“哪些页面已命中 overlay、哪些页面仍待补丁”，不算完整交付。
7. 未区分“代码问题”和“外部依赖问题”，不算合格验收。

## 技能附件

1. 优先复用技能附件中的模板文件，而不是每次从零手写：
   - `assets/Dockerfile.template`
   - `assets/apply_overlay.py.template`
   - `assets/ls-zh.js.template`
   - `assets/ls-zh.css.template`
   - `scripts/extract_label_studio_strings.py`
2. `scripts/extract_label_studio_strings.py` 用于从指定版本源码中全量提取模板分类、模板标题以及模板、前端、配置中的候选 UI 字符串。
3. 这些附件只作为骨架模板，真正的英文文案仍必须来自对应版本源码提取。

## 输出要求

1. 给出采用的 Label Studio 正式版本号和来源链接。
2. 说明哪些页面已经命中 overlay，哪些页面仍需补充公共入口模板补丁。
3. 如果后续要做镜像发布，建议使用“版本分支名 = 版本号，镜像 tag = 同版本号”的方式，但不自动发布，除非用户明确要求。
4. 必须明确说明本次 `ls-zh.js` 是“基于源码全量扫描生成”还是“仅完成注入链路验证”；如果不是前者，视为任务未完成。
