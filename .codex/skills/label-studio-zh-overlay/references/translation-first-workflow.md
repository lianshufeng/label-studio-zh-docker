# Translation-First Workflow

## 目标

1. 把“真实运行时 UI 文案尽可能翻成中文”放在第一位。
2. 不为了维持流程形式而接受高优先级大量漏翻。
3. 不修改官方前端源码，继续通过模板注入和运行时 overlay 完成中文化。
4. 把“换机器、换干净仓库后可重复达到当前中文化水平”作为正式交付目标。

## 交付约定

1. `scripts/` 必须作为正式仓库内容提交，不能只保留在本地。
2. `translations/catalog.json` 必须作为翻译记忆库提交，不能只依赖运行产物反推。
3. `.tmp/` 与 `data/` 可以忽略，但 `scripts/`、`translations/` 不得忽略。

## 数据约定

1. 源码扫描结果必须包含结构化 `entries`，每条至少带：
   - `source`
   - `module`
   - `priority`
   - `occurrences`
   - `kinds`
   - `files`
   - `contexts`
2. 候选词典必须保留：
   - `score`
   - `priority`
   - `status`
   - `contexts`
3. 合并后的翻译记录必须保留：
   - `target`
   - `status`
   - `translation_source`

## 优先级

1. `high`：
   - 登录注册
   - 首页导航
   - 项目创建
   - 导入导出
   - 数据管理主操作
   - 账户设置
   - 热键设置
   - 顶部导航、侧边导航、账户菜单
2. `medium`：
   - 页面说明文案
   - 空状态
   - 帮助提示
   - 模板标题和模板分类
3. `low`：
   - 示例内容
   - 长段说明
   - 低频辅助文本

## 自动补译

1. 先复用 catalog 中的已确认翻译。
2. 再复用内置术语表与短语表做自动补译。
3. 自动补译必须是保守策略：
   - 允许保留 API、JSON、OCR、LLM 等常见英文技术词。
   - 不能留下明显未处理的英文残片。
4. 无法安全自动补译时，保留空 `target`，并进入待补译报告。
5. 对 `source == target` 且目标仍为英文的 catalog 占位项，视为“未翻译”，不得覆盖自动补译结果。

## 去噪

1. 继续排除 docs、tests、stories、fixtures、mocks、e2e。
2. 继续排除代码片段、类名、路径、颜色值、构建常量。
3. 对残句、半句、变量拼接碎片降权或剔除。

## 验收

1. 不能只看是否生成了 `ls-zh.js`。
2. 必须同时检查：
   - 高优先级未翻译数量
   - 运行时未翻译报告
   - 页面抽样是否仍有明显英文按钮、标题、菜单
3. 如果高优先级漏翻仍然很多，只能说明“翻译链已重构”或“注入链路已验证”，不能说“中文化完成”。
4. 焦点页面至少包括：
   - 登录页
   - 首页
   - 数据管理页
   - 项目设置页
   - 账户设置页
   - 热键页
5. 优先使用 `scripts/verify_runtime_overlay.py` 汇总焦点区域必翻短语、页面导向缺口和残留英文，再决定下一轮修复落点。
6. 每次执行都应按“扫描 -> 候选 -> 同步 -> 构建 -> 页面导向验收 -> 分类修复 -> 再生成”循环收敛，而不是只跑一轮。
