# Label Studio 浏览器巡检补译

## 目标

1. 先把当前仓库镜像编译并启动。
2. 再启动浏览器，使用真实页面而不是只看源码做补译。
3. 覆盖主导航、侧边菜单、账号菜单、项目菜单、设置子页、创建项目流程、导入导出、模板页等可达页面。
4. 把可翻译的 UI 文案尽量翻完，只保留明确不应翻译的内容。
5. 最终结果必须落到正式 overlay 文件：
   - `overrides/static/zh-overlay/ls-zh.js`
   - `overrides/static/zh-overlay/ls-zh.css`
6. 每完成一轮翻译后，必须继续回到浏览器重复检查，继续查找残留英文并继续覆盖 `ls-zh.js` 与 `ls-zh.css`，直到整页达到可用的完整中文化水平；只有特殊术语、特殊含义、真实业务数据或明确不应翻译的内容可以保留。

## 适用范围

当用户提出以下诉求时使用本技能：

- “编译并启动后去网页里看还有哪些没翻译”
- “边浏览边翻译”
- “把每个菜单都点进去看”
- “创建项目弹窗、模板页、设置页还有英文”
- “只靠扫源码不够，要按真实页面补”

## 正式产物

必须直接维护仓库内正式文件，而不是只生成临时结果：

1. `overrides/static/zh-overlay/ls-zh.js`
2. `overrides/static/zh-overlay/ls-zh.css`
3. `translations/catalog.json`
4. 如为支撑浏览器巡检和词典生成而必须调整：
   - `scripts/sync_translation_catalog.py`
   - `scripts/build_overlay_js.py`
   - 其他已有 `scripts/*.py`

临时浏览器脚本只允许放在 `/.tmp/`，不得当作正式交付物。

## 执行顺序

### 1. 编译并启动

1. 先在仓库根目录执行：
   - `docker compose up -d --build`
2. 确认容器状态：
   - `docker compose ps -a`
3. 确认登录页可访问：
   - `http://localhost:8090/user/login/`
4. 若启动中出现短时 `ERR_EMPTY_RESPONSE` 或连接被关闭，先看日志，等 `label-studio init` 完成后再继续：
   - `docker compose logs --tail=120 label-studio`

### 2. 启动浏览器并登录

1. 优先使用 Playwright + 本机 Edge/Chrome 做真实页面巡检。
2. 用户若提供账号密码，可在本地临时环境变量中使用；不得写入仓库文件、README、技能正文或最终答复。
3. 登录后必须等待页面稳定，再开始抽取可见文本。

### 3. 菜单覆盖策略

必须按“从可见导航向内层页面逐步展开”的方式巡检，不得只扫固定几个首页路由。

至少覆盖这些入口：

1. 顶部导航
2. 左侧导航
3. 账号头像下拉菜单
4. 项目列表页
5. 项目数据页
6. 项目设置总页
7. 项目设置子页：
   - 常规
   - 标注界面
   - 标注设置
   - 模型
   - 预测
   - 存储
   - Webhooks
   - Danger Zone
8. 创建项目流程：
   - 项目名称
   - 数据导入
   - 标注设置
   - 模板分类页
   - 模板详情或预览
9. 组织/成员页
10. 账户设置页
11. 热键页
12. 导入/导出相关弹窗和菜单

如果页面内还能继续进入下一层子页、标签页、弹窗、抽屉或下拉菜单，继续点进去看。

### 4. 哪些内容该翻译

默认应翻译：

1. 页面标题
2. 菜单名
3. 标签页名
4. 按钮
5. 表单标签
6. 占位符
7. 帮助说明
8. 空状态
9. 弹窗标题和正文
10. 模板分类
11. 模板卡片标题
12. 设置项名称
13. 表格列名
14. 固定提示语
15. 固定营销/引导卡片文案
16. 固定状态文案

默认不翻译，除非明确是官方固定演示文本：

1. 用户自己输入的数据
2. 真实任务正文
3. 标注结果内容
4. 邮箱、用户名、ID、时间戳、日期
5. 代码片段
6. 文件路径
7. 样式类名
8. 真实业务数据载荷

对“动态 UI 文案”允许在 `ls-zh.js` 中补运行时规则，例如：

1. `1 minute ago`
2. `13 minutes ago`
3. `Tasks: 0 / 0`
4. `0 of 0 Tasks (0%)`
5. `Submitted annotations: 0`

## 补译方法

### 1. 优先级

1. 先补浏览器里真实看到的英文
2. 再回源到 `.tmp` 扫描结果和 `translations/catalog.json`
3. 再补 `scripts/sync_translation_catalog.py` 的术语表、内置词条或种子词条
4. 必要时在 `scripts/build_overlay_js.py` 中增加安全的运行时短语规则

### 2. 回写位置

1. 精确词条优先进入 `translations/catalog.json` 及其生成链
2. 高频固定词条优先沉淀到 `scripts/sync_translation_catalog.py`
3. 动态数字/时间类文案可进入 `scripts/build_overlay_js.py`
4. 最终统一重新生成：
   - `overrides/static/zh-overlay/ls-zh.js`
5. 只有确实需要样式修正时才修改：
   - `overrides/static/zh-overlay/ls-zh.css`

### 3. 禁止行为

1. 禁止只根据截图手工改少量文案就宣称完成
2. 禁止只扫首页、登录页、项目列表页
3. 禁止只做词典不做浏览器复查
4. 禁止把账号密码写入仓库
5. 禁止整文件重写正式产物
6. 禁止直接修改官方前端源码

## 每轮迭代都要做的事

1. 浏览器登录并点击更多页面
2. 记录真实残留英文
3. 按类别判断：
   - 词典缺词
   - 动态规则缺失
   - 页面没巡检到
   - 不应翻译
4. 回写到正式生成链
5. 重新执行生成
6. `docker compose up -d --build`
7. 再次浏览器复查
8. 若仍有可翻译残留，继续下一轮，不设“只补一轮”的上限

直到整个页面达到可用的完整翻译状态，或只剩下明确不应翻译的特殊术语、特殊含义、真实数据为止，才能结束。
如果某个页面仍存在成片菜单、标题、按钮、说明、模板名称、设置项或固定提示英文，则必须继续下一轮“浏览器复查 -> 补词/补规则 -> 重建 -> 再复查”，不得提前结束。

## 建议临时脚本

可在 `/.tmp/` 下创建浏览器巡检脚本，例如：

1. 通用登录巡检脚本
2. 创建项目流程专项巡检脚本
3. 项目设置子页专项巡检脚本

这些脚本只用于本轮浏览器抽样和自动点选，不属于正式交付物。

## 结束前检查

结束前必须确认：

1. 镜像已经重新编译并启动
2. 登录页可访问
3. 已做过真实浏览器复查
4. `ls-zh.js` 已回写
5. `ls-zh.css` 已检查，必要时已回写
6. 修改文件为 UTF-8 无 BOM
7. 中文回读未乱码
8. 对已巡检页面再次复查后，只剩下明确可接受的不翻译内容

## 输出要求

最终汇报必须明确说明：

1. 本轮实际浏览了哪些页面
2. 本轮新补了哪些菜单或页面文案
3. 哪些残留还存在
4. 结果已经回写到哪些正式文件
5. 镜像是否已重新编译并启动成功
6. 为什么判定剩余内容属于“特殊术语、特殊含义或不应翻译”
